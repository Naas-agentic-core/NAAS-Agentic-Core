# تشخيص خرافي: لماذا ينهار النظام عند طلب الأدمن حساب عدد ملفات بايثون؟

## 1. الملخص التنفيذي (Executive Summary)
بناءً على طلبكم لعملية تشخيص جراحية دقيقة وخرافية (Super Diagnosis) في ملف واحد فقط، وبدون المساس بأي سطر برمجي (Zero Code Modification)، نُقدم هذا التقرير الجوهري للإجابة على سؤال واحد: **لماذا تظهر كارثة `Error connecting to agent: All connection attempts failed` عندما يطلب مدير النظام (Admin) "كم عدد ملفات بايثون في المشروع؟"**

المشكلة الحقيقية ليست غباءً في وكيل الذكاء الاصطناعي، وليست خللاً في كود العد نفسه، بل هي ضريبة مباشرة للانتقال المعماري العملاق من بنية متجانسة (Monolith) إلى بنية الخدمات المصغرة بنسبة 100% (Microservices).

## 2. التشريح الجذري للمشكلة (The Root Cause Anatomy)

### العصر القديم (Monolith Era):
في النظام القديم، كان الخادم (Server) يعمل كوحدة واحدة. عندما يسأل الأدمن عن عدد الملفات، كان النظام يلتقط النية وينفذ أمر الشل مباشرة في نفس البيئة:
`find . -name "*.py" | wc -l`
كانت العملية تتم محلياً في الخلفية، تعود النتيجة فوراً، ولا يوجد أي اتصال شبكي (No Network Hopping) يمكن أن يفشل.

### العصر الحديث (100% Microservices API-First Era):
استجابةً للقواعد المعمارية الصارمة الجديدة (LangGraph, MCP, DSPy, TLM, Multi-Agent)، تم تقسيم النظام:
1. **واجهة الأدمن وبوابة API Gateway:** تعمل في مكان.
2. **العقل المنفذ `orchestrator-service`:** يعمل كخدمة مصغرة معزولة تماماً في مكان آخر.

**ماذا يحدث بالضبط عندما يطلب الأدمن عد الملفات؟**
1. كاشف النوايا (المعزز بـ DSPy) يلتقط السؤال ويصنفه بنجاح فائق كـ `ADMIN_QUERY`.
2. بدلاً من التنفيذ المحلي، يُجبر النظام (بناءً على التوجيه الإلزامي) على إرسال الطلب عبر شبكة HTTP إلى خدمة المنسق `orchestrator-service`.
3. يستخدم عميل الواجهة مكتبة `httpx` للاتصال برابط مثل: `http://orchestrator-service:8006/agent/chat`.
4. **لحظة الكارثة (The Catastrophe):** إذا كان عميل الواجهة (النواة أو الـ Gateway) يعمل خارج الشبكة الافتراضية للخدمة المصغرة (مثلاً: خارج Docker Network كـ `cogniforge-network`)، أو إذا لم يتم حقن متغير البيئة `ORCHESTRATOR_SERVICE_URL` بشكل صحيح ليوجه إلى منفذ الـ `localhost` المطلوب، يفشل نظام حل الأسماء (DNS Resolution) في العثور على `orchestrator-service`.
5. ينتهي هذا بانهيار مكتبة الاتصال وتوليد الاستثناء الصارخ: `httpx.ConnectError: All connection attempts failed`.
6. يعترض النظام هذا الخطأ الشبكي الخام، ويغلفه كـ JSON، ثم يرسله عبر الـ WebSocket للواجهة كحدث `assistant_error`، لتظهر الرسالة حرفياً للمستخدم:
`Error connecting to agent: All connection attempts failed`.

## 3. لماذا لم يحدث هذا في السابق؟
لأننا في السابق لم نكن نحتاج لعبور الشبكة من أجل تنفيذ أمر بسيط كهذا. التفكيك الشبكي (Network Decoupling) أضاف طبقة هشاشة (Network Fragility)؛ فإذا انقطع الاتصال بين مكونات النظام الموزع، تفشل حتى أبسط العمليات وتظهر كأخطاء في الواجهة، وهذا طبيعي تماماً في عالم الـ Microservices قبل إحكام طبولوجيا الشبكة (Network Topology).

## 4. الخلاصة التشخيصية المذهلة
النظام لم يفقد ذكاءه ولا قدرته على عد الملفات. أداة العد `count_python_files` موجودة ومبرمجة بأعلى مستوى داخل الـ `orchestrator-service` وتنتظر عبر الـ MCP.
**السبب الوحيد** للخطأ هو فشل الاتصال الشبكي (Connectivity Regression) بين العميل والخدمة. مجرد ضبط شبكة Docker أو تصحيح مسارات الخدمة كفيل بحل هذه المعضلة وإعادة النظام لعمله الخرافي.
