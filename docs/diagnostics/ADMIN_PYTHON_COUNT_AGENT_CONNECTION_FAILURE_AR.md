# التشريح الجراحي الشامل: لماذا فشل الآن أمر "كم عدد ملفات بايثون؟"

## 1) ملخص تنفيذي شديد الدقة

العطل الحالي **ليس** خللاً في عملية العد نفسها، بل خلل **مسار توجيه واتصال** بعد التحول من Monolith إلى Microservices:

1. سؤال الأدمن من نوع "كم عدد ملفات بايثون" يُصنَّف Intent = `ADMIN_QUERY`.
2. هذا التصنيف يُجبر `ChatOrchestrator` على تفويض الطلب إلى `orchestrator-service` عبر HTTP streaming (`/agent/chat`).
3. عند فشل الوصول الشبكي للخدمة (DNS/Port/Service down)، يلتقط العميل الاستثناء ويعيد رسالة موحدة:  
   `Error connecting to agent: All connection attempts failed`
4. الواجهة تعرض الرسالة كما هي مع بادئة `Error:` فيظهر النص الذي شاهدته حرفياً.

**النتيجة:** المشكلة حدثت لأن طبقة "العد" لم تعد تُنفَّذ محلياً مباشرةً (كما في النمط المونوليثي السابق)، بل أصبحت تعتمد على مسار شبكة ميكروسيرفس إلزامي.

---

## 2) الدليل الجنائي (Evidence Chain)

### الدليل A: تصنيف سؤال "عدد ملفات بايثون" كـ Admin Query
- Registry يحتوي أنماطاً عامة جداً مثل:
  - `(python|بايثون|\.py)`
  - `(files|ملفات|ملف|عدد\s*ملفات)`
  - `(كم|count|احسب)\s*(عدد)?\s*(ملف|بايثون|python)`
- لذلك أي سؤال مثل "كم عدد ملفات بايثون" يدخل مباشرة في `ADMIN_QUERY`.

### الدليل B: كل `ADMIN_QUERY` يُفوض إلى وكيل ميكروسيرفس
- في `ChatOrchestrator.process()` يتم اعتبار `ADMIN_QUERY` ضمن `is_agent_intent`.
- هذا المسار يستدعي `orchestrator_client.chat_with_agent(...)` بدل التنفيذ المحلي.

### الدليل C: نقطة الفشل الحقيقية هي الاتصال بالخدمة
- `orchestrator_client.chat_with_agent()` يعمل `POST` على:
  - `http://<orchestrator-base-url>/agent/chat`
- عند أي استثناء اتصال/شبكة، يُعاد حدث:
  - `{"type":"assistant_error", "payload":{"content":"Error connecting to agent: <exception>"}}`
- وبذلك يصبح `<exception>` = `All connection attempts failed`.

### الدليل D: لماذا كان النظام قديماً "يعد بدقة خارقة"؟
- سابقاً كانت الطلبات الإدارية تُنفذ داخل نفس العملية (Monolith/Local tooling) أو مسار داخلي قريب من shell/filesystem.
- الآن صارت نفس النية مرتبطة بجسر HTTP بين خدمات مستقلة؛ أي فقدان خدمة/شبكة = فشل مباشر قبل الوصول لأداة العد نفسها.

---

## 3) الأسباب الجذرية (Root Causes) مرتبة حسب التأثير

## RC-1 (حرج): كسر الانعزال عن طريق "اعتماد شبكي إلزامي" لأوامر إدارية بسيطة
- أمر بسيط مثل "count python files" لم يعد محلياً؛ أصبح Remote Delegation إجباري.
- هذا يضيف نقطة فشل جديدة (availability coupling) لم تكن موجودة بنفس الحدة في المونوليث.

## RC-2 (حرج): عدم وجود Fallback محلي عندما يفشل Orchestrator
- عند فشل الاتصال، المسار لا يحاول تنفيذ بديل Local Admin Path.
- النتيجة: فشل كامل حتى لعمليات يمكن نظرياً تنفيذها محلياً.

## RC-3 (عالٍ): أنماط Intent واسعة جداً (Over-broad Regex)
- وجود أنماط عامة مثل `ملف` أو `python` يلتقط استفسارات كثيرة ويحوّلها تلقائياً لمسار الوكيل البعيد.
- هذا يزيد احتمال المرور عبر مسار شبكي هش لأوامر كان يمكن أن تبقى في مسار داخلي.

## RC-4 (عالٍ): افتراضات بيئة التشغيل في URL الافتراضي
- `ORCHESTRATOR_SERVICE_URL` يُحل افتراضياً إلى `http://orchestrator-service:8006` خارج حالة Codespaces.
- إذا التطبيق يعمل خارج شبكة Docker الداخلية (أو DNS الاسم غير معروف)، سيحدث `ConnectError` فوراً.

## RC-5 (متوسط): صياغة الخطأ النهائية لا تعطي جذر المشكلة التشغيلي
- الرسالة المعروضة للمستخدم صحيحة تقنياً لكنها سطحية تشغيلياً.
- لا توضّح إن كانت المشكلة: DNS / service down / network segmentation / wrong env URL.

---

## 4) لماذا هذا متسق مع مبادئ Microservices رغم الألم الحالي؟

نعم، المعمارية API-first microservices تتطلب:
- استقلالية الخدمات,
- اتصال HTTP/gRPC,
- عدم الاعتماد على shared internals.

لكن تطبيق هذا التحول على عمليات Admin بسيطة **بدون آليات مرونة (fallback, health-gated routing, circuit-aware policy)** خلق فجوة تشغيلية:
- المعمارية صحيحة مبدئياً،
- لكن Reliability Envelope غير مكتمل لمسار الأدمن.

---

## 5) تشخيص احتمالات العطل التشغيلية الأكثر واقعية الآن

1. **Orchestrator service غير شغّال** (Crash/Not started/Health failing).
2. **اسم المضيف `orchestrator-service` غير قابل للحل** من بيئة تشغيل النواة الحالية.
3. **المنفذ 8006 غير متاح** (binding/firewall/reverse proxy mismatch).
4. **متغير ORCHESTRATOR_SERVICE_URL غير مناسب للبيئة الفعلية**.
5. **تعارض مسارات بين gateway/orchestrator** يجعل الاستدعاء المباشر غير صالح في هذا النشر.

---

## 6) الخلاصة الحاسمة

الذي "تعطل" ليس عد ملفات بايثون، بل **سلسلة التفويض الشبكي** التي أصبحت شرطاً أساسياً لتنفيذ الطلب الإداري بعد الانتقال المعماري.

بعبارة واحدة:
> **انتقلتم من Local Capability إلى Remote Capability بدون طبقة أمان تشغيلية مكافئة، فظهر خطأ الاتصال مكان نتيجة العد.**

---

## 7) توصيات علاجية (تشخيصية/تشغيلية) بدون تعديل شيفرة الآن

1. توثيق Runbook إلزامي لمسار Admin Query:
   - فحص صحة `orchestrator-service` قبل قبول طلبات ADMIN_QUERY.
2. ضبط مصفوفة نشر واضحة:
   - Docker internal DNS vs localhost mapping لكل بيئة.
3. إضافة مؤشرات مراقبة:
   - نسبة فشل `/agent/chat`، زمن الاستجابة، وأسباب ConnectError مصنفة.
4. مراجعة قواعد Intent كحوكمة تشغيلية:
   - تقليل الالتقاط العام لعبارات `ملف/python` غير الدقيقة.
5. اعتماد سياسة Degradation رسمية:
   - عند فقدان الوكيل البعيد: رسالة تشغيلية دقيقة + مسار fallback محكوم.

> تم إعداد هذا الملف كتشخيص جنائي عميق فقط، بدون أي تعديل على الشيفرة التنفيذية.
